1. PURPOSE OF THIS DOCUMENT
===============================================================================

This TRD converts the approved Product Requirements Document into an exact technical implementation plan for data engineering, feature engineering, machine learning, anomaly detection, predictive modeling, explainability, Streamlit dashboard development, testing, deployment, and documentation.

The system must be technically strong, visually professional, interpretable to non-technical stakeholders, and honest about the limitations of the supplied data.

The expected evaluation target is a high-quality submission equivalent to approximately 9/10. Reaching that standard depends on:

- Correct use of the supplied dataset
- Leakage-safe time-aware modeling
- Appropriate use of bagging, boosting, and stacking
- Accurate and honest evaluation
- Machine-specific baseline modeling
- Strong anomaly detection
- Actionable predictive-maintenance insights
- Clear business context for every major visualization
- Plain-language explanations for non-technical users
- Reproducibility, testing, and deployment quality
- Avoidance of unsupported failure-prediction claims

The project must not chase model complexity only to appear advanced. Every algorithm must be retained only when it produces measurable, time-aware validation improvement or operational value.

===============================================================================
2. CRITICAL DATA LIMITATION AND MODELING POSITION
===============================================================================

The supplied dataset contains:

- 100,000 observations
- 50 machines
- 14 source columns
- Active, Idle, and Maintenance operation modes
- Temperature, vibration, power, network, production, quality, error, and maintenance-related signals
- Efficiency_Status values of High, Medium, and Low

The supplied dataset does not contain:

- Confirmed machine failure timestamps
- Breakdown labels
- Failure component labels
- Remaining useful life
- Downtime duration
- Maintenance work-order IDs
- Repair actions
- Replacement records
- Downtime cost
- Verified cause-and-effect labels

Therefore, the technical solution must use a dual analytical architecture:

TRACK A — UNSUPERVISED MACHINE-HEALTH ANOMALY ENGINE
Detects abnormal machine behavior without requiring failure labels.

TRACK B — SUPERVISED FUTURE-DETERIORATION PROXY ENGINE
Predicts a carefully constructed future operational-deterioration proxy using bagging, boosting, and stacking models.

The supervised model must never be described as a confirmed failure model. Its output must be named one of:

- Future Deterioration Probability
- Proxy Maintenance Risk Probability
- Early-Warning Proxy Probability

It must not be named:

- Failure Probability
- Breakdown Probability
- Remaining Useful Life
- Exact Failure Time

A third optional analytical track may classify the current Efficiency_Status for benchmarking, but this is secondary and must not replace predictive maintenance.

3. TECHNICAL GOALS
===============================================================================

3.1 Core Goals

- Build a reproducible data pipeline for the supplied CSV.
- Learn machine- and operation-mode-specific normal behavior.
- Generate leakage-safe time-dependent features.
- Detect multivariate anomalies.
- Predict future operational deterioration through supervised ensembles.
- Use bagging, boosting, and stacking correctly.
- Calibrate model probabilities.
- Fuse anomaly, deterioration, trend, persistence, and operational-impact signals.
- Assign Low, Medium, and High risk.
- Rank machines by maintenance priority.
- Explain every significant alert.
- Provide operational recommendations.
- Build an interactive Streamlit application.
- Make all charts understandable to technical and non-technical users.
- Export reports and machine-risk tables.
- Provide tests, model cards, documentation, and deployment files.

3.2 Accuracy Goals

“Most accurate” must mean the best model under time-aware validation and operational metrics, not the highest random-split accuracy.

Primary model-selection metrics:

- PR-AUC for future deterioration
- Recall of deterioration events
- Precision at top-K alerts
- F2 score
- Matthews correlation coefficient
- Balanced accuracy
- Calibration error
- False alerts per machine per day
- Proxy early-warning lead time
- Risk ranking stability
- Performance stability by machine and operation mode

Secondary metrics:

- ROC-AUC
- Accuracy
- Macro F1
- Weighted F1
- Log loss
- Brier score

Raw accuracy must not be used as the only or primary model-selection metric because the target may be imbalanced.

8. TIME INDEX AND SAMPLING REQUIREMENTS
===============================================================================

The global dataset is minute-indexed, but individual machines are observed irregularly because records are distributed across machines.

Therefore:

- Never assume that consecutive rows for one machine are one minute apart.
- Never calculate rolling windows across unsorted data.
- Use Machine_ID grouping before lags and rolling calculations.
- Prefer timestamp-aware rolling windows.
- Record time_since_previous_observation.
- Use a minimum-observation requirement for rolling calculations.
- Mark insufficient-history rows.
- Never backfill from future observations.
- Use forward-only calculations.

Supported time windows should be configurable, for example:

- 1 hour
- 6 hours
- 12 hours
- 24 hours
- 72 hours
- 7 days

Observation-count fallback windows may include:

- 3 observations
- 5 observations
- 10 observations
- 20 observations

The selected production windows must be justified using the actual machine observation density.

9.2 Safe Ratio Formula

All ratios must use:

ratio = numerator / max(abs(denominator), epsilon)

Use a configurable epsilon.
Replace infinite values with null.
Impute or flag only after leakage-safe fitting.

9.3 Baseline Fitting Rule

All baseline statistics used in validation and test rows must be learned only from:

- The training period
or
- Past observations available before the row timestamp

Global calculations using the entire dataset are prohibited for model features.

9.4 Categorical Encoding

CatBoost:
- Pass categorical values natively where appropriate.

LightGBM/XGBoost/scikit-learn:
- Use OneHotEncoder or target-independent ordinal encoding.
- Do not use target encoding unless implemented out-of-fold within training only.
- Treat Machine_ID as categorical.
- Avoid excessive machine memorization.
- Compare performance with and without Machine_ID.

9.5 Scaling

Required for:

Preferred:
- RobustScaler for heavy-tailed sensor deviations

Not required for:
- Random Forest
- Extra Trees
- XGBoost
- LightGBM
- CatBoost

The scaler must be fitted on training data only and serialized.

10. TARGET AND PROXY LABEL DESIGN
===============================================================================

10.1 Why a Proxy Target Is Required

The dataset lacks confirmed failure events. A future-deterioration target must be constructed transparently using future operational behavior.

10.2 Primary Proxy Target

Name:
future_deterioration_proxy

Type:
Binary

At timestamp t for a machine, the label becomes 1 when a meaningful deterioration event occurs within the selected future horizon.

The primary label should be based on a composite event rather than current Efficiency_Status alone.

10.3 Candidate Future Deterioration Conditions

A future row or future-window aggregate may contribute one event point when it meets a training-derived condition:

- Error rate reaches an extreme machine-mode percentile and rises materially from current state.
- Defect rate reaches an extreme percentile and rises materially.
- Production speed falls to a low percentile and declines materially.
- Temperature and vibration jointly enter extreme ranges.
- Power instability rises while production falls.
- A transition occurs from High or Medium efficiency to Low.
- The machine enters Maintenance mode after sustained abnormal behavior.
- Predictive_Maintenance_Score declines materially, used only in a secondary label experiment.

10.4 Recommended Composite Rule

A primary deterioration event should require at least two independent adverse conditions within the future horizon.

Example:

future_deterioration_proxy = 1 when any two or more of the following occur:

1. future_error_rate >= training machine-mode P90
2. future_defect_rate >= training machine-mode P90
3. future_production_speed <= training machine-mode P10
4. future_vibration_deviation >= robust threshold
5. future_temperature_deviation >= robust threshold
6. future Efficiency_Status transitions from High/Medium to Low
7. future Maintenance mode occurs following persistent abnormality

This prevents a single noisy signal from defining the target.

Exact thresholds must be derived from training data only and finalized after label-quality analysis.

10.5 Prediction Horizons

Build separate labels for:

- 6-hour deterioration
- 24-hour deterioration
- 72-hour deterioration

Retain the horizon that provides:

- Sufficient positive examples
- Useful early-warning time
- Stable performance
- Operational relevance

A multi-horizon dashboard may be retained if all models remain reliable.

10.6 Label Leakage Protection

At time t, model inputs may use only information available at or before t.

The label may inspect the future horizon, but:

- No future aggregate may be present in features.
- No backward fill from future values is allowed.
- No label threshold may be fitted on validation or test data.
- Rows near the dataset end without a full future horizon must be excluded or marked censored.
- Future Maintenance mode cannot appear as a current feature.
- Future Efficiency_Status cannot appear as a feature.

10.7 Secondary Targets

Optional current-state benchmark:
- Efficiency_Status multiclass classification

Optional ordinal benchmark:
- Low, Medium, High encoded ordinally

These models are for comparison and dashboard context only. They are not the main predictive-maintenance model.

11. MODELING ARCHITECTURE
===============================================================================

11.1 Track A — Unsupervised Anomaly Detection

Mandatory model:
- Isolation Forest

Optional models:
- Autoencoder
- PCA reconstruction error
- ECOD or HBOS through PyOD
- Local Outlier Factor for offline analysis
- One-Class SVM on a reduced sample

Recommended production anomaly ensemble:

- Isolation Forest score
- Autoencoder reconstruction score, only if it adds value
- Robust multivariate deviation score
- Persistence-adjusted anomaly score

The anomaly engine must detect behavior that differs from machine and mode baselines.

11.2 Track B — Supervised Proxy Deterioration Prediction

Mandatory candidate families:

Bagging:
- Random Forest
- Extra Trees

Boosting:
- XGBoost
- LightGBM
- CatBoost
- HistGradientBoostingClassifier

Linear baseline:
- Logistic Regression with class weighting

Simple baseline:
- Dummy classifier
- Rule-based risk baseline

Stacking:
- Time-aware out-of-fold stacked ensemble

Optional blending:
- Weighted probability blend of best diverse models

11.3 Why Each Family Is Included

Random Forest:
- Robust bagging baseline
- Handles nonlinear interactions
- Stable
- Provides feature importance
- Good benchmark

Extra Trees:
- Strong variance reduction
- More randomized splits
- Often competitive on tabular data
- Useful diversity for stacking

XGBoost:
- Strong gradient boosting
- Good regularization
- Handles nonlinear interaction
- Suitable for imbalanced classification

LightGBM:
- Efficient on large tabular data
- Strong accuracy
- Supports class weighting
- Useful categorical and missing-value behavior

CatBoost:
- Strong treatment of categorical variables
- Reduced encoding complexity
- Often effective with machine and mode categories
- Provides SHAP-compatible explanations

HistGradientBoosting:
- Fast scikit-learn baseline
- Useful dependency-light fallback

Logistic Regression:
- Interpretable reference model
- Useful meta-learner
- Calibration-friendly

11.4 Model Selection Principle

Do not assume stacking is automatically best.

The production model should be:

- Best validated single model
or
- Best calibrated blend
or
- Best stacked model

Select the simplest option that meets or exceeds the operational performance of more complex alternatives.

- class_weight
- bootstrap: False and optional True comparison

12.3 Bagging Evaluation

Evaluate:

- Stability across time folds
- Probability calibration

===============================================================================
14. STACKING AND BLENDING REQUIREMENTS
===============================================================================

14.1 Time-Aware Out-of-Fold Predictions

Stacking must use walk-forward out-of-fold predictions.

Incorrect:
- Train base models on all training data.
- Predict the same training data.
- Train meta-model on those predictions.

Correct:
1. Divide training period into ordered folds.
2. Train each base model on earlier folds.
3. Predict the immediately later fold.
4. Combine all out-of-fold predictions.
5. Train meta-model on out-of-fold predictions only.
6. Retrain base models on the full development period.
7. Generate test predictions.
8. Pass test predictions to the fitted meta-model.

14.2 Recommended Base Models

Use diverse models, not five nearly identical boosters.

Recommended stack:

- Extra Trees
- LightGBM
- CatBoost
- XGBoost
- Logistic Regression baseline
- Optional anomaly score as an additional meta-feature

14.3 Meta-Model

Preferred:
- Regularized Logistic Regression

Alternative:
- Small LightGBM model with strict complexity controls

The meta-model should remain simple to reduce overfitting.

14.4 Stacking Inputs

- Calibrated or raw base probabilities
- Anomaly score
- Risk-persistence score
- Model disagreement statistics
- Optional operation-mode indicators

14.5 Blending Alternative

A weighted blend may be selected when it is more stable than stacking.

Example:

proxy_probability =
    w1 * lightgbm_probability
  + w2 * catboost_probability
  + w3 * extra_trees_probability
  + w4 * xgboost_probability

Weights must:

- Sum to one
- Be optimized on validation data only
- Be constrained to prevent extreme reliance on one model
- Be documented
- Be tested across time folds

14.6 Ensemble Retention Rule

Keep the stack or blend only if it produces a meaningful improvement in at least one of:

- PR-AUC
- Recall at fixed precision
- Precision at top-K
- Calibration
- Temporal stability
- Alert lead time
- Lower false-alert burden

If improvement is negligible, use the best simpler model.

===============================================================================

Expected techniques:

- Class weights
- scale_pos_weight
- Balanced bootstrap samples
- Threshold tuning
- Focal loss only if justified and supported
- Precision-recall evaluation

SMOTE rules:

===============================================================================
17. TEMPORAL SPLITTING AND BACKTESTING
===============================================================================

17.1 Minimum Split Design

Example only; exact dates must be determined from data:

- Train: earliest 60%
- Validation: next 20%
- Test: latest 20%

Preferred:
- Multiple expanding-window folds

Example:

Fold 1:
Train A → Validate B

Fold 2:
Train A+B → Validate C

Fold 3:
Train A+B+C → Validate D

Final:
Train development period → Test latest held-out period

17.2 Split Requirements

- No random shuffle for primary evaluation.
- All preprocessing fitted within each fold.
- All thresholds fitted within each fold.
- All proxy-label percentiles fitted from training data.
- All calibration fitted using out-of-sample development predictions.
- Final test period remains untouched until model selection is complete.

17.3 Machine Coverage

Confirm each split contains sufficient observations per machine.
If not:

- Report unsupported machines.
- Use global fallback behavior.
- Do not fabricate machine-specific metrics.

17.4 Optional Generalization Test

Hold out selected machines to test cross-machine generalization.

This is a secondary experiment and must be clearly separated from chronological deployment evaluation.

- Reliability diagram
- Log loss

19. ANOMALY ENGINE TECHNICAL SPECIFICATION
===============================================================================

19.1 Isolation Forest

Inputs:
- Core numeric sensor signals
- Machine-mode deviation features
- Rolling instability features
- Interaction features
- Network features
- Exclude Efficiency_Status
- Exclude Predictive_Maintenance_Score in the primary model

Preprocessing:
- Robust scaling optional
- Remove constant features
- Impute training medians only
- Winsorization only if values are confirmed valid but numerically destabilizing; do not erase true anomalies

Candidate parameters:

- n_estimators: 300 to 1200
- contamination: auto plus validation-derived candidates
- max_samples: 256 to full sample
- max_features: 0.5 to 1.0
- bootstrap: False and optional True
- random_state fixed

19.2 Autoencoder

Optional only.

Architecture example:
- Input
- Dense 128
- Dense 64
- Latent 16 or 32
- Dense 64
- Dense 128
- Output

Requirements:
- Robust-scaled input
- Train on a normal-behavior subset
- Dropout or L2 regularization if needed
- Early stopping
- Validation reconstruction monitoring
- Per-feature reconstruction error
- CPU-compatible model size
- Serialized model and scaler
- Compare against Isolation Forest

Do not retain if:
- It adds no measurable detection benefit.
- It substantially increases deployment complexity.
- Explanations become materially weaker.

19.3 Anomaly Score Normalization

For each model:

- Orient score so higher = more abnormal.
- Use training score distribution.
- Apply robust percentile or empirical CDF transformation.
- Produce 0–100 normalized score.
- Clip only at display bounds.
- Preserve raw score.

19.4 Anomaly Ensemble

Example:

anomaly_ensemble_score =
    0.60 * isolation_forest_score
  + 0.25 * autoencoder_score
  + 0.15 * robust_deviation_score

If no autoencoder:

anomaly_ensemble_score =
    0.75 * isolation_forest_score
  + 0.25 * robust_deviation_score

Weights must be validated rather than accepted blindly.

===============================================================================
20. FINAL RISK FUSION ENGINE
===============================================================================

20.1 Inputs

- Normalized anomaly score
- Calibrated future-deterioration proxy probability
- Persistence score
- Trend-escalation score
- Operational-impact score
- Model-agreement score
- Data-quality confidence

20.2 Example Risk Formula

base_risk =
    0.35 * anomaly_score_normalized
  + 0.35 * proxy_probability_percent
  + 0.15 * persistence_score
  + 0.10 * trend_escalation_score
  + 0.05 * operational_impact_score

Then adjust:

- Increase evidence when anomaly and supervised models agree.
- Reduce evidence when model disagreement is high.
- Reduce confidence when history is insufficient.
- Flag data-quality concern when network or sensor data is unreliable.
- Do not hide the individual component scores.

The exact production weights must be chosen through validation and expert-readable reasoning.

20.3 Risk Levels

Initial defaults:

Low:
- final_risk_score < 60

Medium:
- 60 <= final_risk_score < 80

High:
- final_risk_score >= 80

Alternative:
- Percentile-based thresholds by machine and operation mode

Final thresholds must optimize:

- High-risk precision
- Deterioration recall
- False alerts per machine per day
- Alert workload
- Lead time
- Threshold stability

20.4 Persistence Overrides

Examples:

- Three or more consecutive Medium observations may escalate to High.
- One transient High observation may be marked “Transient High” rather than “Persistent High.”
- Rapid score growth can trigger escalation before the absolute threshold.
- Insufficient history must prevent overconfident escalation.

20.5 Evidence Strength

High evidence:
- Multiple models agree
- Multiple signals contribute
- Risk persists
- Trend is increasing
- Data quality is good

Medium evidence:
- Model agreement but limited persistence
or
- Persistent anomaly with moderate supervised probability

Low evidence:
- Single-model alert
- Single-signal spike
- Insufficient history
- Data-quality issue
- Strong model disagreement

45. FINAL TECHNICAL POSITIONING
===============================================================================

FactoryGuard 6G will not be a basic efficiency-classification dashboard.

It will be a hybrid predictive-maintenance intelligence system combining:

- Machine-aware baseline modeling
- Multivariate anomaly detection
- Bagging models
- Gradient-boosting models
- Time-aware stacking or blending
- Probability calibration
- Future-deterioration proxy prediction
- Risk persistence
- Temporal escalation
- Operational impact scoring
- Explainable maintenance prioritization
- Business-readable Streamlit analytics

The strongest possible model is not automatically the most complicated model. The final production algorithm must be the model or ensemble that demonstrates the best combination of:

- Predictive quality
- Early-warning value
- Low false-alert burden
- Temporal stability
- Calibration
- Interpretability
- Deployment practicality

The final system must clearly communicate:

Detect
→ Predict proxy deterioration
→ Validate persistence
→ Explain contributors
→ Prioritize machines
→ Recommend inspection
→ Compare post-maintenance behavior
→ Monitor model and data quality

This architecture satisfies the product requirements while preserving analytical honesty and providing the advanced machine-learning depth expected from a high-scoring project.
