Never call it confirmed failure probability.

STRICT LEAKAGE RULES

Prevent all forms of target and temporal leakage.

The following must not be used as inputs to the primary anomaly model:

* Efficiency_Status
* Predictive_Maintenance_Score

For the supervised proxy model:

* Do not use future observations as features.
* Do not backfill from future values.
* Do not calculate rolling statistics using future rows.
* Do not fit scalers, encoders, baselines, thresholds, or imputers on validation or test data.
* Do not use random train-test splitting as the primary evaluation.
* Do not apply SMOTE before time-based splitting.
* Do not train stacking models using in-sample base-model predictions.
* Do not tune models on the final test period.

STACKING REQUIREMENTS

Implement stacking correctly:

1. Divide the development period into ordered temporal folds.
2. Train base models only on earlier data.
3. Predict the immediately later fold.
4. Combine these out-of-fold predictions.
5. Train the meta-model only on out-of-fold predictions.
6. Retrain base models on the complete development period.
7. Generate final test predictions.
8. Pass those predictions through the trained meta-model.

Preferred diverse base models:

* Extra Trees
* LightGBM
* CatBoost
* XGBoost
* Logistic Regression

Preferred meta-model:

* Regularized Logistic Regression

Do not assume stacking is automatically the best model.

The production model must be selected from:

* Best single model
* Best calibrated blended model
* Best correctly implemented stacked model

Retain the ensemble only if it provides meaningful improvement.

HYPERPARAMETER OPTIMIZATION

Use Optuna.

Requirements:

* Time-aware validation folds
* Trial pruning
* Reproducible random seeds
* Saved Optuna study
* Early stopping for boosting models
* No tuning on the final test set
* Separate development and final test evaluation
* Save all trial results

Use approximately:

* 20–40 trials per model during development
* 75–150 trials for the strongest final candidates where compute permits

CLASS IMBALANCE

Prefer:

* Class weights
* scale_pos_weight
* Balanced bootstrap sampling
* Threshold tuning
* Precision-recall metrics

Use SMOTE only if it demonstrates improvement and only inside training folds.

Do not alter the natural class distribution of validation or test data.

TIME-AWARE VALIDATION

Use chronological evaluation.

Preferred:

* Expanding-window validation
* Walk-forward validation
* Latest period reserved as untouched final test set

All feature transformations, baselines, thresholds, encoders, imputers, and scalers must be fitted separately inside each training fold.
